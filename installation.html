1
SSH Mac → Ubuntu
Accès distant depuis le Mac de travail
✓ Documenté
Étape 1 — Sur Ubuntu · installer le serveur SSH

Ubuntu 24.04 Desktop n'inclut pas openssh-server par défaut. À faire une seule fois, directement sur la machine.

# Installer openssh-server
sudo apt update && sudo apt install -y openssh-server

# Activer le démarrage automatique et démarrer maintenant
sudo systemctl enable --now ssh

# Vérifier que le service tourne
sudo systemctl status ssh
  # → résultat attendu :
  Active: active (running) since ...
Étape 2 — Sur Ubuntu · hostname mDNS (recommandé Freebox Pop)

Avec une Freebox Pop, l'IP locale peut changer (DHCP). La solution recommandée est d'utiliser le hostname mDNS (jean-mi.local) — stable sans configuration réseau, supporté nativement par macOS (Bonjour) et Ubuntu (Avahi).

# Vérifier le hostname Ubuntu
hostname
  → jean-mi  (ou le nom choisi lors de l'installation)

# Définir le hostname si besoin (remplacer jean-mi par ton choix)
sudo hostnamectl set-hostname jean-mi

# Vérifier qu'avahi-daemon (mDNS) est actif
sudo systemctl status avahi-daemon
  → Active: active (running) ...

# Installer avahi si absent (rare sur Ubuntu 24.04)
sudo apt install avahi-daemon
✓ Une fois fait, le PC est joignable via jean-mi.local depuis le Mac (et tout appareil du réseau local). Utiliser ce nom à la place de IP_UBUNTU dans toutes les commandes suivantes.
⚠️ Remplacer USER par ton vrai nom d'utilisateur Ubuntu dans les commandes suivantes.
Étape 3 — Sur Mac · générer une clé Ed25519 dédiée

Créer une clé SSH spécifique à cette connexion (ne pas réutiliser la clé GitHub).

ssh-keygen -t ed25519 -C "mac → jean-mi ubuntu" -f ~/.ssh/jean_mi_ubuntu
  → Generating public/private ed25519 key pair.
  → Passphrase : laisser vide ou choisir une passphrase
Étape 4 — Sur Mac · copier la clé publique vers Ubuntu
ssh-copy-id -i ~/.ssh/jean_mi_ubuntu.pub USER@jean-mi.local
  → /usr/bin/ssh-copy-id: INFO: 1 key(s) were left to be installed.
# Saisir le mot de passe Ubuntu une dernière fois
Après cette étape, la connexion par mot de passe n'est plus nécessaire.
Étape 5 — Sur Mac · configurer ~/.ssh/config

Ajouter ce bloc dans ~/.ssh/config (créer le fichier si absent).

Host jean-mi
  HostName jean-mi.local
  User USER
  IdentityFile ~/.ssh/jean_mi_ubuntu
  AddKeysToAgent yes
  UseKeychain yes
Si tu as choisi un hostname différent, remplacer jean-mi.local par ton-hostname.local. Le .local est obligatoire pour la résolution mDNS.
Étape 6 — Sur Mac · tester la connexion
ssh jean-mi
  # → résultat attendu :
  Welcome to Ubuntu 24.04 LTS ...
  USER@nipogl:~$
✓ Connexion sans mot de passe — la phase SSH est terminée.

Docker Engine
Prérequis pour Home Assistant, Mosquitto et Zigbee2MQTT
✓ Documenté
Installer Docker via le repo officiel apt

Ne pas utiliser le paquet snap ni le paquet Ubuntu — utiliser le repo officiel Docker.

# Supprimer d'éventuelles anciennes versions
sudo apt remove -y docker docker-engine docker.io containerd runc 2>/dev/null; true

# Dépendances
sudo apt update && sudo apt install -y ca-certificates curl lsb-release

# Clé GPG officielle Docker (format .asc — méthode recommandée depuis 2025)
sudo install -m 0755 -d /etc/apt/keyrings
sudo curl -fsSL https://download.docker.com/linux/ubuntu/gpg \
  -o /etc/apt/keyrings/docker.asc
sudo chmod a+r /etc/apt/keyrings/docker.asc

# Ajouter le repo
echo \
  "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.asc] \
  https://download.docker.com/linux/ubuntu \
  $(lsb_release -cs) stable" \
  | sudo tee /etc/apt/sources.list.d/docker.list > /dev/null

# Installer Docker Engine + Compose
sudo apt update && sudo apt install -y docker-ce docker-ce-cli containerd.io docker-compose-plugin

# Ajouter l'utilisateur au groupe docker (évite sudo)
sudo usermod -aG docker $USER
newgrp docker

# Vérifier l'installation Docker
docker run --rm hello-world
  # → résultat attendu :
  Hello from Docker!

Home Assistant Container
Déploiement via Docker Compose · accès sur :8123
✓ Documenté
Créer le répertoire et le fichier compose
mkdir -p ~/homeassistant && cd ~/homeassistant
cat > compose.yaml << 'EOF'
services:
  homeassistant:
    image: ghcr.io/home-assistant/home-assistant:stable
    container_name: homeassistant
    restart: unless-stopped
    network_mode: host
    privileged: true
    volumes:
      - ./config:/config
      - /etc/localtime:/etc/localtime:ro
      - /run/dbus:/run/dbus:ro
EOF
Démarrer Home Assistant
docker compose up -d
  → [+] Running 1/1
  → ✔ Container homeassistant  Started

# Suivre les logs au démarrage
docker compose logs -f
  → (attendre "Home Assistant initialized")
Accéder à HA depuis le réseau local : http://jean-mi.local:8123
Ou depuis le Mac via tunnel SSH : ssh -L 8123:localhost:8123 jean-mi puis ouvrir http://localhost:8123
4
Dongle Zigbee
SONOFF ZigBee 3.0 USB Dongle Plus (EFR32MG21) · udev · dialout
✓ Documenté
Étape 1 — Identifier le port série après branchement USB

Brancher le dongle, puis repérer son chemin dans /dev/serial/by-id/ — ce chemin est stable entre les redémarrages, contrairement à /dev/ttyACM0.

⚠️ Le ZBDongle-E (EFR32MG21) utilise la puce USB WCH CH9102F (driver cdc-acm) et crée un /dev/ttyACM0 — pas un /dev/ttyUSB0 comme la version CC2652P.
# Lister les ports ACM (CDC-ACM — ZBDongle-E)
ls /dev/ttyACM* 2>/dev/null

# Confirmer la détection dans dmesg
dmesg | tail -20
  # → résultat attendu :
  usb 1-1: cdc_acm 1-1:1.0: ttyACM0: USB ACM device

# Trouver le by-id stable — à utiliser dans Z2M
ls -la /dev/serial/by-id/
  # → résultat attendu :
  usb-Itead_Sonoff_Zigbee_3.0_USB_Dongle_Plus_V2_XXXXX-if00-port0 -> ../../ttyACM0
Copier le chemin complet /dev/serial/by-id/usb-Itead_Sonoff_… — il sera utilisé dans le compose.yaml de Zigbee2MQTT.
Étape 2 — Ajouter l'utilisateur au groupe dialout

Sans cela, accéder au port série nécessite sudo. Docker le passera en privileged, mais c'est bon à avoir pour les tests manuels.

sudo usermod -aG dialout $USER
newgrp dialout

# Vérifier
groups
  → bertrand adm dialout cdrom sudo ...
⚠️ Fermer et rouvrir la session pour que le groupe prenne effet globalement.
Étape 3 (optionnel) — Règle udev pour alias permanent

Crée un alias /dev/zigbee-sonoff indépendant du numéro attribué au boot. Vendor/Product ID du WCH CH9102F : 1a86:55d4.

# Créer la règle udev (idVendor/idProduct WCH CH9102F — ZBDongle-E EFR32MG21)
sudo tee /etc/udev/rules.d/99-sonoff-zigbee.rules << 'EOF'
SUBSYSTEMS=="usb", ATTRS{idVendor}=="1a86", ATTRS{idProduct}=="55d4", SYMLINK+="zigbee-sonoff", GROUP="dialout", MODE="0666"
EOF

# Recharger les règles
sudo udevadm control --reload-rules && sudo udevadm trigger

# Vérifier
ls -la /dev/zigbee-sonoff
  → /dev/zigbee-sonoff -> ttyACM0

Mosquitto + Zigbee2MQTT
Broker MQTT + bridge Zigbee · Docker Compose · interface web :8080
✓ Documenté
Étape 1 — Créer la structure de répertoires
mkdir -p ~/homeassistant/mosquitto/{config,data,log}
mkdir -p ~/homeassistant/zigbee2mqtt/data
Étape 2 — Fichier mosquitto.conf
cat > ~/homeassistant/mosquitto/config/mosquitto.conf << 'EOF'
listener 1883
protocol mqtt
allow_anonymous true
persistence true
persistence_location /mosquitto/data/
EOF
Étape 3 — Fichier configuration.yaml pour Zigbee2MQTT
cat > ~/homeassistant/zigbee2mqtt/data/configuration.yaml << 'EOF'
version: 5
frontend:
  enabled: true
  port: 8080
homeassistant:
  enabled: true
mqtt:
  base_topic: zigbee2mqtt
  server: "mqtt://mosquitto:1883"
serial:
  port: /dev/ttyACM0
  adapter: auto
permit_join: true
advanced:
  network_key: GENERATE
  pan_id: GENERATE
  channel: 15
EOF
Le port /dev/ttyACM0 correspond au mapping devices: du compose.yaml ci-dessous (ZBDongle-E EFR32MG21).
Étape 4 — Étendre compose.yaml avec Mosquitto + Zigbee2MQTT

Ajouter ces deux services dans ~/homeassistant/compose.yaml, après le service homeassistant. Remplacer le chemin du device par le by-id trouvé en Phase 4.

  mosquitto:
    image: eclipse-mosquitto:latest
    container_name: mosquitto
    restart: unless-stopped
    ports:
      - "1883:1883"
    volumes:
      - ./mosquitto/config:/mosquitto/config
      - ./mosquitto/data:/mosquitto/data
      - ./mosquitto/log:/mosquitto/log

  zigbee2mqtt:
    image: koenkk/zigbee2mqtt:latest
    container_name: zigbee2mqtt
    restart: unless-stopped
    privileged: true
    depends_on:
      - mosquitto
    ports:
      - "8080:8080"
    environment:
      TZ: Europe/Paris
    volumes:
      - ./zigbee2mqtt/data:/app/data
      - /run/udev:/run/udev:ro
    devices:
      - /dev/serial/by-id/usb-Itead_Sonoff_Zigbee_3.0_USB_Dongle_Plus_V2_XXXXX-if00-port0:/dev/ttyACM0
Étape 5 — Démarrer et accéder à l'interface
cd ~/homeassistant && docker compose up -d

# Suivre les logs Z2M (vérifier la connexion MQTT + coordinateur)
docker compose logs -f zigbee2mqtt
  → Coordinator firmware version: {"meta":{"maintrel":3,"majorrel":2,...}}
  → Connected to MQTT server
  → Started frontend on port 8080
Interface Zigbee2MQTT : http://jean-mi.local:8080
⚠️ Intégration MQTT dans HA : HA Container utilise network_mode: host (réseau de l'OS), donc il ne peut pas résoudre le nom mosquitto. Lors de la configuration de l'intégration MQTT dans HA (Settings → Devices & Services → MQTT), utiliser localhost comme broker (port 1883) — pas mosquitto:1883.
Étape 6 — Appairer les appareils Zigbee

permit_join: true est déjà actif dans la config. Mettre l'appareil en mode appairage :

NOUS A1Z (prise)
Maintenir le bouton 5–7 secondes jusqu'au clignotement rapide du voyant → Z2M détecte l'appareil automatiquement.
SONOFF ZBMINIL2 (plafonnier)
Maintenir le bouton 5 secondes (voyant s'éteint) — OU basculer l'interrupteur mural 10 fois rapidement.
# Suivre les logs pendant l'appairage
docker compose logs -f zigbee2mqtt
  → Successfully interviewed 'NOUS A1Z', model 'A1Z'
  → Pairing successful
Étape 7 — Désactiver permit_join après appairage
# Éditer la config Z2M
sed -i 's/permit_join: true/permit_join: false/' ~/homeassistant/zigbee2mqtt/data/configuration.yaml

# Redémarrer Z2M
docker compose restart zigbee2mqtt
⚠️ Garder permit_join: false en usage normal — évite l'appairage non sollicité d'appareils tiers.

ReSpeaker XVF-3800
4 mics · AEC · beamforming · plug & play Linux
✓ Documenté
Étape 1 — Vérifier la détection USB

Le XVF-3800 est plug & play sur Linux via le driver USB Audio standard — aucune installation de driver requise.

# Vérifier la présence du périphérique USB
lsusb
  → Bus 001 Device 003: ID 2886:0018 ... reSpeaker XVF3800

# Lister les cartes audio détectées (noter le numéro de carte X)
arecord -l
  → card X: Array [reSpeaker XVF3800 4-Mic Array], device 0: USB Audio [USB Audio]
Retenir le numéro de carte X — il sera utilisé dans toutes les commandes suivantes à la place de CARD.
Étape 2 — Test d'enregistrement

Le canal 0 (left) contient le flux traité (AEC + beamforming) — c'est celui à utiliser pour la reconnaissance vocale.

# Enregistrer 5 secondes (remplacer CARD par le numéro trouvé)
arecord -D plughw:CARD,0 -c 2 -r 16000 -f S16_LE -d 5 test.wav

# Écouter l'enregistrement
aplay test.wav
Étape 3 — Définir le ReSpeaker comme carte ALSA par défaut
# Créer ~/.asoundrc (remplacer CARD et SPEAKER_CARD par vos numéros)
cat > ~/.asoundrc << 'EOF'
pcm.!default {
  type asym
  playback.pcm "speaker"
  capture.pcm "mic"
}
pcm.mic {
  type plug
  slave { pcm hw:CARD,0; rate 16000 }
}
pcm.speaker {
  type plug
  slave { pcm hw:SPEAKER_CARD,0 }
}
ctl.!default {
  type hw
  card CARD
}
EOF

# Vérifier avec un enregistrement simple
arecord -d 3 -f S16_LE -r 16000 verif.wav && aplay verif.wav
⚠️ Remplacer CARD par le numéro de la carte ReSpeaker, et SPEAKER_CARD par le numéro de la sortie audio (HDMI, jack…) — visible dans aplay -l.

Voix locale — Whisper + Piper + Wyoming
STT + TTS + wake word · tout en Docker · HA Assist en français
✓ Documenté
Architecture

HA Container n'a pas d'add-on store. Les services vocaux tournent en tant que conteneurs Docker séparés, connectés à HA via le protocole Wyoming (TCP).

ServiceImage DockerPort WyomingRôle
wyoming-whisperrhasspy/wyoming-whisper10300Transcription vocale (STT)
wyoming-piperrhasspy/wyoming-piper10200Synthèse vocale française (TTS)
wyoming-openwakewordrhasspy/wyoming-openwakeword10400Détection mot de réveil
Étape 1 — Ajouter les services dans compose.yaml

Modèle Whisper conseillé : tiny-int8 pour commencer (rapide), puis base-int8 en usage courant. Le Ryzen 7 7735HS gère facilement small-int8.

  wyoming-whisper:
    image: rhasspy/wyoming-whisper:latest
    container_name: wyoming-whisper
    restart: unless-stopped
    ports:
      - "10300:10300"
    volumes:
      - ./whisper-data:/data
    command: --model tiny-int8 --language fr
    environment:
      TZ: Europe/Paris

  wyoming-piper:
    image: rhasspy/wyoming-piper:latest
    container_name: wyoming-piper
    restart: unless-stopped
    ports:
      - "10200:10200"
    volumes:
      - ./piper-data:/data
    command: --voice fr_FR-siwis-medium
    environment:
      TZ: Europe/Paris

  wyoming-openwakeword:
    image: rhasspy/wyoming-openwakeword:latest
    container_name: wyoming-openwakeword
    restart: unless-stopped
    ports:
      - "10400:10400"
    volumes:
      - ./openwakeword-data:/data
    environment:
      TZ: Europe/Paris
    # Pas de "command" — les modèles par défaut (ok_nabu, hey_jarvis, alexa) se chargent automatiquement
    # Note : --preload-model a été supprimé en v2.0.0 (oct. 2025)
Étape 2 — Démarrer les services
# Créer les répertoires de données (modèles téléchargés automatiquement)
mkdir -p ~/homeassistant/{whisper-data,piper-data,openwakeword-data}

# Démarrer tous les services
cd ~/homeassistant && docker compose up -d

# Suivre le téléchargement des modèles (2–5 minutes au 1er démarrage)
docker compose logs -f wyoming-whisper wyoming-piper wyoming-openwakeword
  → Loaded model tiny-int8
  → Loaded voice fr_FR-siwis-medium
  → Loaded wake word model ok_nabu
Étape 3 — Connecter Wyoming à Home Assistant

Dans HA : Settings → Devices & Services → Add Integration → Wyoming Protocol

Ajouter 3 fois (une par service) :

HostPortService reconnu
jean-mi.local10300Whisper STT
jean-mi.local10200Piper TTS
jean-mi.local10400openWakeWord (ok nabu)
Ensuite : Settings → Voice Assistants → modifier l'assistant par défaut → Language : Français · STT : Wyoming (Whisper) · TTS : Wyoming (Piper) · Wake word : Wyoming (ok nabu)

Spotify + YouTube
Spotify Connect via API native HA · YouTube via yt-dlp + mpv
✓ Documenté
8A — Spotify · Créer une app Developer

Un compte Spotify Premium est obligatoire depuis février 2026. L'intégration native HA utilise OAuth2.

  1. Aller sur developer.spotify.com/dashboard → Create app
  2. App name : Jean-Mi Home Assistant
  3. Dans Edit Settings → Redirect URIs → ajouter exactement : https://my.home-assistant.io/redirect/oauth
  4. Copier le Client ID et le Client Secret
8A — Spotify · Intégrer dans Home Assistant

Settings → Devices & Services → Add Integration → Spotify
Saisir Client ID + Client Secret → suivre le flux OAuth → une entité media_player.spotify_USERNAME est créée.

Lancer une playlist via un service HA (dans Developer Tools → Services) :

service: media_player.play_media
target:
  entity_id: media_player.spotify_USERNAME
data:
  media_content_type: playlist
  media_content_id: "spotify:playlist:XXXXXXXXXXXX"
Obtenir l'URI d'une playlist : Spotify app → ··· → Share → Copy Spotify URI → format spotify:playlist:XXXX
8A — Spotify · Commandes vocales en français

Settings → Automations & Scenes → Create Automation → Trigger : Sentence

# Exemples de phrases à configurer
Joue ma playlist de travail
Joue la playlist cozy
Lance Spotify

Action : appeler media_player.play_media avec l'URI de la playlist correspondante.

8B — YouTube · Installer yt-dlp + mpv

Solution recommandée pour Ubuntu sans Chromecast : mpv + yt-dlp.

# Installer mpv et yt-dlp (yt-dlp disponible dans noble/universe — pas besoin de pip)
sudo apt install -y mpv yt-dlp

# Créer la config mpv
mkdir -p ~/.config/mpv
cat > ~/.config/mpv/mpv.conf << 'EOF'
ytdl-format=best[height<=720]/best
volume=100
EOF

# Test direct
mpv "https://www.youtube.com/watch?v=VIDEO_ID"
8B — YouTube · Intégrer dans Home Assistant

Dans ~/homeassistant/config/configuration.yaml :

shell_command:
  youtube_play: "nohup mpv '{{ url }}' &"

Puis redémarrer HA et appeler le service depuis une automation ou Developer Tools :

service: shell_command.youtube_play
data:
  url: "https://www.youtube.com/watch?v=VIDEO_ID"
